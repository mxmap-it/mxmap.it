# Hidden Cloud Backends Behind Italian Gateways

> Generated by `scripts/analyze_hidden_backends.py` from `data.json` (22,987 IT entities).

**Detection logic (in priority order):**

1. **DKIM CNAME** → `*.onmicrosoft.com` (MS365) / `google._domainkey.*` (Google) / `amazonses` (AWS)
2. **SPF include** → `spf.protection.outlook.com` / `_spf.google.com` / `amazonses`
3. **Provider field** already set to microsoft/google/aws (ground truth)
4. **`cloud_tenant_only` field** (set by upstream DKIM CNAME detector for 219 entries)

**Key finding:**

Italian-origin private PA SaaS gateways (gecomail, epublic, sitek, halley, host-it, vianova, etc.) typically **do not publish DKIM records** — only 0-3% of their users have any DKIM. This means the DKIM-based detection can only find a tiny fraction of hidden backends for these gateways. **The vast majority of suspected hidden-backend entries cannot be confirmed via passive DNS analysis alone.** Active probing (ANAC procurement cross-reference, bounce-probe, SMTP banner verification) is needed to confirm.

## 6. Methodology — Why Hidden-Backend Counts Are Conservative

The numbers above are **lower bounds**. The 1,200-1,500 estimate from `reports/gateway_ita_analysis.md` (June 2026) was based on the assumption that *high `independent` rate = hidden cloud backend*. This analysis shows that assumption is **wrong** for Italian PA SaaS gateways: they don't publish DKIM, so we can't see what's behind the local MX.

**What's actually happening:**

1. Italian PA SaaS gateways (gecomail, epublic, halley, vianova, cliocom, antispamsolution, widestore, demosdata) handle ~440+ customers each with **0 published DKIM records**.
2. The MX is a local server. The actual mailbox backend (often MS365 or Google Workspace) is reached via secure SMTP relay — invisible to passive DNS.
3. The 0-3% with DKIM show their real backend (`onmicrosoft.com` for MS365 users), confirming the pattern but covering a tiny fraction of the population.

**To get a true hidden-backend count, you need to:**

- **ANAC procurement cross-reference** (suggested by Fpietrosanti / Nuke): match the entity's `name`/`cf` against Italian public IT procurement contracts in the last 3 years (open data: `dati.anticorruzione.it/opendata`). Contracts for "Microsoft 365", "Google Workspace", or "cloud mail" for the same entity are strong evidence of the hidden backend.
- **Bounce probe** (`scripts/bounce_probe.py`): send a test email to each entity's public PEC / contatti address and inspect the DSN / bounce headers — the `Received:` chain reveals the actual delivery path (Italian gateway → MS365 / Google / self-hosted).
- **SMTP banner verification**: connect to the MX port 25 and read the `220` banner — many M365/Google installations leak the upstream provider name even when the MX hostname is local.

Until one of these active methods is run, the table in §1 shows the **provable** hidden-backend count, which is much smaller than the suspected total.
